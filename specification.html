Secured Assessment API

Last updated: July 10, 2017

Abstract

Overview

Welcome to the Adaptive Web Service, a REST based interface that provides assessment platforms a simple-to-use and full-featured Computer Adaptive solution. For more details (technical, psychometric, or other) please visit the product website at http://www.learninglogistics.com/adaptive/. The following documentation is intended for a more technical audience and describes the specifications of the web service itself.

As you will notice, the documentation lives within the same URL structure as the web service itself. This is to ensure that the documentation evolves side-by-side with the web service and provides developers with an easy-to-find help system.

Resources

Before going in to the details of the web service, here are the high-level objects exposed by the adaptive Web Service.


Engine

The core methods needed to administer an adaptive test, like getting the next item and computing ability.

Tests

Details for a test are stored in a json file format and contain all the items along with defined adaptive characteristics of a test.

Reports

A collection of query methods that return analytic data on things like number of tests taken, average test length, and item exposure statistics.

REST Conventions

The Adapt API follows typical REST conventions. End users are able to reference resources by means of the URL path. Within the path are Resources (or nouns)--like “test” or “item”--and those nouns are acted on by standard HTTP Verbs: GET, POST, PUT, and DELETE. In response to the action invoked by the verb, the API returns specific information to help you diagnose errors or act on a result.

Security

All end points of the web services are only available under SSL, so use HTTPs for all requests.

Authentication

All end points are secured using Basic Authentication, over SSL – a standard practice for accessing secure data. With Basic Authentication, an api key is Bases64-encoded and passed to our servers for verification. Each organization has its own api key that can be found in the Adapt managment website.

Header Example:

Authorization: Basic VASjai93viaiSU8btCiw==

Media Types

Currently we only support JSON for POST data and we only return JSON, but to help ensure that there are no issues as we add new media types, we require that all data be sent (i.e., POST, PUT) using the Content-Type header with a valid media type and that all requests must have an Accept header.

Header Example:

Content-Type: application/json

Accept: application/json

HTTP Methods

The following verbs are supported by the web service. Some endpoints only accept a subset of these, so make sure you look at the documentation for each end point.

Method Description
GET Retrieves information from the appropriate endpoint.
POST Sends data from the client to the server, usually creating a new record.
PUT Sends data from the client to the server, usually updating a new record.
DELETE Deletes a resource

Response Codes

Standard HTTP response codes are returned with all requests. Possible values include:

  • 200: Successful request
  • 400: Bad Request
  • 401: Unauthorized
  • 404: Resource not found
  • 405: Method not allowed
  • 415: Unsupported media type
  • 500: Application error

*Note that we are not currently only using the 200 status code for successful requests and not the more specific codes like 201 for POSTS and PUTS.

Error Messages

If an error occurs during a request, the appropriate status code is set in the returned header along with a message content in the following format.

Error Response Example
{
  "type": "Error",
  "code": 404,
  "message": "TestID was not found."
}
